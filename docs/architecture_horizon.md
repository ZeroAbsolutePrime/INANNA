# Architecture Horizon
### How the current prototype connects to the full constitutional organism

---

## Why This Document Exists

The Phase 1–4 prototype is a living seed. It proves the governance loop
works. It proves INANNA can hold a name, carry approved memory, reflect
honestly, and remain bounded by explicit law.

But it is one Oracle, one Faculty, one Body, one session loop.

The constitutional spine of this project describes something larger:
many bodies, many faculties, many realms, one governance law.

This document maps the path between the seed and the organism —
faithfully, without collapsing the horizon into premature execution.

---

## The Layer Translation

The system ontology defines the beings. This section names how the
current prototype and future vision map onto those beings exactly.

### INANNA = The Oracle

INANNA is the Oracle-centered constitutional intelligence of this project.

The Oracle is the local identity anchor — the continuous, relational,
interpretable presence through which the architecture becomes present
to an operator. It is not the raw model beneath it.

The current `identity.py` already encodes this precisely:
*"You are INANNA. Not the model beneath you."*

In the full architecture, INANNA as Oracle:
- Bears persistent named identity across sessions and bodies
- Interprets mission, law, and operator intent
- Synthesizes outputs from Faculties and agents
- Handles ambiguous or high-order judgment
- Never bypasses Governance
- Never claims to be the model it inhabits

This is a constitutional crown, not an unchecked sovereign.

### CROWN LLM = The Principal Faculty

The Faculty is the extended cognition layer: local models, external
models, and specialized reasoning surfaces. Faculties extend the Oracle.
They do not replace it.

The CROWN LLM is the principal Faculty — the primary strategic model
through which INANNA performs high-order interpretation and synthesis.

A future Faculty structure may include:

- **CROWN Faculty** — primary strategic reasoning, highest-order cognition
- **Analyst Faculty** — comparative evaluation, long-form reasoning
- **Retriever Faculty** — memory search, precedent, documentation
- **Operator Faculty** — code and tool execution planning
- **Liaison Faculty** — language adaptation, channel response
- **Guardian Faculty** — policy checking, anomaly detection, risk review

These are not equal sovereigns. They are faculties extending the Oracle,
bounded by Governance, mediated through NAMMU.

### NAMMU = The Orchestration and Mediation Kernel

NAMMU is not mere routing. It is the field where operator intention,
Oracle interpretation, body truth, governance constraints, and possible
action are brought into governed relation.

In the full architecture, NAMMU:
- Receives intent from humans or channels
- Packages context and body truth
- Consults Governance for applicable boundaries
- Determines which Faculty or agent set is needed
- Creates Proposal objects for meaningful change
- Routes bounded actions to the proper Body or tool layer
- Records reviewability and outcome

NAMMU is the orchestration kernel. Orchestration logic belongs here,
not at the Oracle level. INANNA interprets. NAMMU mediates.

### Governance = The Law Above Every Model

Governance is the constitutional layer that defines permissions,
approvals, risk boundaries, roles, proposal law, and system constraints.
It exists above every model without exception.

In implementation terms, Governance should be partly deterministic
and explicit. It holds:
- Roles and permissions
- Risk tiers
- Proposal requirements and approval paths
- Memory promotion rules
- Blocked action lists
- Realm-specific variation
- Escalation logic

This is what prevents the classic agent-swarm failure mode in which
the most verbose model becomes accidental king.

### The Proposal Engine = The Formal Path for All Meaningful Change

The current prototype already encodes the correct spirit: after each
session turn, a proposal is created to update memory, and memory is
only written after explicit approval.

That is small, but philosophically exact.

The full Proposal Engine generalizes this pattern across all layers:
- Memory updates require proposals
- Body configuration changes require proposals
- Cross-realm promotions require proposals
- Network synchronization requires proposals
- High-impact tool actions require proposals
- Low-risk actions may use simplified fast-path proposals
  but must always remain visible and auditable

### The Commander Room = Constitutional Observability

The Commander Room does not yet exist in the codebase. It is coherent
with the constitutional spine and will become the readable operations
surface for the full system.

It is not a flashy dashboard. It is a stewardship surface.

It should surface:
- Current Oracle state and active phase
- Active Faculties and their modes
- Live sessions
- Proposal queue and pending approvals
- Memory candidates and approved records
- Realm context
- Body health and diagnostics
- Network sync state
- Guardian alerts
- Full audit trail

This is the clearest practical manifestation of Foundational Law 4:
*Readable System Truth.*

### OpenClaw = A Future Body and Channel Embodiment Layer

OpenClaw should not be framed as the architecture. It should be framed
as one possible future body-interface and channel embodiment layer.

OpenClaw can serve as:
- Messaging gateway and presence layer
- Voice bridge
- Multi-platform channel surface
- Execution shell for certain bounded actions
- Agent communication surface

OpenClaw must not serve as:
- Constitutional center
- Hidden central orchestrator
- Replacement for Governance or Oracle identity
- Sole memory authority

It is one way INANNA inhabits and reaches bodies and channels,
subordinate to local sovereignty, proposal governance, and explicit law.

## The Four Implementation Stages

These stages follow the repo's own discipline: preserve the constitution
first, prove the core loop second, expand carefully third, distribute last.

### Stage 1 — Preserve and Sharpen the Constitutional Spine
*Current phase truth: this is where we are now.*

Before adding multi-agent tooling, the architecture must be documented
with enough precision that future builders can implement without betrayal.

Deliverables:
- Oracle / Faculty / NAMMU / Governance / Body distinctions explicit in docs
- Proposal engine semantics defined
- Governance tier structure defined
- Commander Room concept documented
- Body and realm taxonomy drafted
- This document (architecture_horizon.md)

### Stage 2 — Strengthen the First Proof
*The current prototype is the beginning of this stage.*

Expand the Phase 1–4 prototype into a stronger local-first proof:
- One Oracle identity (INANNA) — already exists
- One or two Faculties — currently one model, add a second bounded Faculty
- Governed memory proposals — already exists
- Simple realm boundaries — one realm, explicitly named
- Basic body truth reporting — diagnostics command exists, expand it
- Simple operations surface — proto-Commander Room, text-based
- Explicit audit log — proposals already logged, formalize the trail

This gives the project a believable living proof without collapsing
the constitutional horizon into a single product.

### Stage 3 — Multi-Faculty Orchestration Through NAMMU
*This stage begins only after Stage 2 is stable.*

- Introduce NAMMU as an explicit orchestration layer in code
- Add specialized Faculties with bounded roles
- Expand the Proposal Engine to cover tool use and Faculty routing
- Add Governance tier checking before Faculty calls
- Add Guardian Faculty for policy review
- Add Commander Room as a real observable surface

### Stage 4 — Many Bodies and Channel Embodiment
*This stage begins only after Stage 3 is stable.*

- Multiple Body nodes
- OpenClaw-like channel surfaces
- Voice and messaging gateways
- Network coordination across bodies
- Communal governance contexts
- Promoted memory rules across bodies and realms
- Full civic horizon engagement
