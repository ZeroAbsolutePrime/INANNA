## The Structure

Development is organized in **Cycles of nine phases each**.

Each Cycle is one complete breath of the system:
- Nine phases of bounded, governed, tested building
- One Code Doctrine update grounded in what was learned
- One Cycle Completion Record declaring what was proven
- One handoff document naming what comes next

Nine phases per Cycle is not arbitrary. It mirrors the constitutional
nine-phase structure of the first proof and provides enough granularity
to stay precise without fragmenting into noise.

**Codex always works inside one phase at a time.**
**The Command Center always holds the full Cycle map.**
**The Guardian always holds the full horizon.**

## The Unchanging Rules Across All Cycles

These rules apply from Cycle 1 to Cycle 5. They do not relax
as the system grows. They deepen.

**Codex reads before it builds.**
`ABSOLUTE_PROTOCOL.md`, `CURRENT_PHASE.md`, `CODEX_DOCTRINE.md` —
in that order — before every session, without exception.

**One phase at a time.**
Codex never looks ahead. It never begins the next phase without
a new `CURRENT_PHASE.md` from the Command Center.

**Nine phases per Cycle.**
Each Cycle ends with a completion record and a Code Doctrine update.
Each Cycle is a complete proof, not a partial one.

**The law layer is never touched by implementation agents.**
Documents in `docs/` that are marked as constitutional layer
may only be modified by the Command Center with Guardian approval.
Code agents build. They do not govern.

**Ontological names are reserved until their phase.**
Oracle, NAMMU, Faculty, Body, Governance — these names are not
available for use until the phase document that introduces them
has been written by the Command Center and authorized by the Guardian.

**The proposal engine applies to everything.**
As the system expands — tool use, Faculty routing, cross-body sync,
OS-level changes — the proposal engine expands with it. No meaningful
state change ever happens silently. This rule holds from Phase 1
of Cycle 1 to Phase 9 of Cycle 5.
