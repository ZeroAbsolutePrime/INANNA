# Future Handoff

This repository exists so INANNA NYX can survive across time without conceptual collapse.

## What Future Builders Must Preserve

Future builders must preserve:

- the working project sentence
- the foundational laws
- the distinction between Oracle, Faculty, Body, Governance, Proposal, and Network
- governance above the model
- proposal before change
- local sovereignty as the center of gravity
- plurality without fragmentation
- truthful limits instead of premature product theater

## What Future Builders May Change

Future builders may change:

- runtime stack
- model providers
- operating systems
- storage and memory implementation
- embodiment strategy
- network architecture
- deployment form
- UI and interface surfaces

The implementation may evolve freely if the deeper truths remain intact.

## What Must Not Happen

Future work must not:

- collapse the project into a generic AI assistant
- treat governance as a permissions afterthought
- let the model become the sovereign layer
- centralize coordination into hidden ownership
- confuse the first proof with the whole architecture
- import legacy material without explicit framing

## Recommended Builder Sequence

Before implementation, a future builder should:

1. Preserve the law layer.
2. Preserve the ontology.
3. Clarify the first believable manifestation.
4. Define governance movements before action surfaces.
5. Translate only then into technical architecture.

## Handoff Principle

The purpose of this repo is not to freeze the future. It is to make the future buildable without betrayal.
