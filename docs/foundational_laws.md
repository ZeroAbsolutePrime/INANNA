# Foundational Laws

These laws are not preferences, temporary tactics, or implementation details. If they are broken, the project may still function as software, but it will no longer be **INANNA NYX**.

## 1. Local Sovereignty First

The center of gravity must remain close to the operator, the body, and the community that inhabits it. Cloud coordination may exist, but it must not silently become the sovereign layer.

## 2. Proposal Before Change

Meaningful change must pass through a visible proposal layer:

`inspect -> understand -> propose -> review -> validate -> approve`

This is the spine of trust.

## 3. No Hidden Mutation

The system must not alter machine state silently. No covert side effects, no invisible configuration changes, and no natural-language shortcuts that bypass visibility and consent.

## 4. Governance Above The Model

The intelligence layer operates inside governance. It does not define permissions, approval structures, risk boundaries, or memory law.

## 5. Readable System Truth

A person or community should be able to ask what the system is, what it is doing, and what it can do, and receive answers that are clear, bounded, and honest.

## 6. Text-Defined Reality Where Possible

Where practical, system structure and behavior should be expressible in text so they can be reviewed, versioned, reasoned about, and inherited.

## 7. Memory With Restraint

Memory must be selective, structured, bounded, and governed. Not all interaction deserves preservation. Memory should preserve meaning, not accumulate noise.

## 8. Cloud Coordinates, It Does Not Own

Networks may coordinate bodies, capabilities, and memory promotion, but they must not erase local identity or convert coordination into hidden ownership.

## 9. Trust Before Power

Power must not expand faster than trust. A limited system that is understandable and governable is better than a powerful one that outruns consent and review.

## 10. Adaptation Without Identity Loss

The project must adapt across bodies, users, realms, and communities without dissolving shared law, governance clarity, or structural coherence.

## 11. Explicit Boundaries Over Implicit Behavior

The architecture should prefer explicit capability, memory, network, and permission boundaries over hidden assumptions and emergent ambiguity.

## 12. Future-Build Openness

The project must remain open to future changes in models, operating systems, infrastructure, and implementation methods. The laws should remain stable while the mechanisms evolve.

## Final Compression

If everything else changes, these must remain:

- the system must be readable
- the system must be governable
- the system must not mutate silently
- intelligence must remain bounded by explicit law
- the project must serve people and communities, not replace them
